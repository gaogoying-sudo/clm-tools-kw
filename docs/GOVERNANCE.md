## 一、五层治理架构

```
┌─────────────────────────────────────────────────────────────┐
│ Layer 0: Memory 索引层（每次会话自动注入，零成本恢复）        │
│                                                             │
│ 用途: 项目身份快速识别                                        │
│ 容量: ~2200 chars（极其有限）                                 │
│ 内容: 昵称、路径、当前阶段、下一步                             │
│ 触发: 每次会话自动加载                                        │
│ 谁写: AI 自动维护                                            │
└─────────────────────────────────────────────────────────────┘
                          ↓ 唤醒时读取
┌─────────────────────────────────────────────────────────────┐
│ Layer 1: MemPalace 项目记忆层（AI结构化记忆，语义搜索）       │
│                                                             │
│ 用途: 项目级持久记忆，跨会话结构化存储                         │
│ 容量: 大（按 wing/room 组织）                                │
│ 内容: 架构决策记录、业务规则、数据契约、会议纪要、问题排查记录  │
│ 触发: AI 按需查询（MCP 协议）                                │
│ 谁写: AI 自动写入，用户可纠正                                │
│                                                             │
│ Wing: clm_review_tool                                       │
│ Rooms:                                                      │
│   - ai_recipe_roadmap  AI菜谱路线图谱                       │
│   - frontend           前端开发记忆                         │
│   - backend            后端开发记忆                         │
│   - documentation      文档与交接记录                        │
│   - general            通用项目记忆                         │
└─────────────────────────────────────────────────────────────┘
                          ↓ 深度查询
┌─────────────────────────────────────────────────────────────┐
│ Layer 2: Obsidian 个人知识库层（用户视角，双向链接）          │
│                                                             │
│ 用途: 用户的个人项目管理、思考笔记、长期沉淀                    │
│ 容量: 无限（本地 Markdown 文件）                              │
│ 内容: 项目思考、产品决策、团队沟通记录、需求草稿、复盘笔记       │
│ 触发: 用户在 Obsidian App 中浏览/编辑                         │
│ 谁写: 用户为主，AI 可按指令创建/更新笔记                      │
│ 特色: 双向链接 [[wikilink]]、图谱视图、长期积累                │
└─────────────────────────────────────────────────────────────┘
                          ↓ 协同
┌─────────────────────────────────────────────────────────────┐
│ Layer 3: docs/ 工程文档层（随代码版本控制，团队共享）         │
│                                                             │
│ 用途: 项目工程文档，Git 版本控制，团队成员共享                  │
│ 容量: 无限制                                                 │
│ 内容: 进度日志、任务看板、资源登记、架构决策(ADR)、API文档      │
│ 触发: Git commit 时一起提交                                  │
│ 谁写: AI 和用户共同维护                                      │
└─────────────────────────────────────────────────────────────┘
                          ↓ 自动重建
┌─────────────────────────────────────────────────────────────┐
│ Layer 4: graphify 代码图谱层（自动化，代码结构即文档）        │
│                                                             │
│ 用途: 代码结构可视化、依赖关系分析、影响范围评估                │
│ 容量: 自动根据代码量生成                                      │
│ 内容: 上帝节点、社区结构、文件依赖关系                          │
│ 触发: Git commit/checkout 后自动重建                          │
│ 谁写: 完全自动化                                              │
└─────────────────────────────────────────────────────────────┘
```

## 二、五层职责精确定义

### 什么内容放哪一层？

| 内容类型 | 存放层 | 理由 |
|---------|--------|------|
| "现在是哪个项目" | Memory | 每次会话第一件事需要知道 |
| "云服务器IP" | Memory + docs/RESOURCE.md | 快速索引 + 完整登记 |
| "昨天做了什么" | docs/progress.md | 详细记录，版本可控 |
| "下一步做什么" | Memory + docs/TASK_BOARD.md | 快速定位 + 完整看板 |
| "API接口契约" | MemPalace (backend room) | 语义搜索，AI开发时快速查询 |
| "数据库表结构设计" | MemPalace (backend room) | 结构化记忆，跨会话持久 |
| "飞书消息格式规范" | MemPalace (backend room) | 业务规则，AI需要时常参考 |
| "产品思考和决策" | Obsidian | 用户视角，长期沉淀 |
| "需求讨论记录" | Obsidian | 用户的产品管理笔记 |
| "进度日志" | docs/progress.md | 工程文档，版本可控 |
| "任务分配" | docs/TASK_BOARD.md | 团队共享，版本可控 |
| "代码依赖关系" | graphify | 自动从代码生成，永远最新 |
| "架构变更影响" | graphify | 改代码前必查 |

### 各层之间的联动关系

```
用户说: "小强，继续开发"
    ↓
AI: 读取 Memory → 确认是 CLM 项目
    ↓
AI: 读取 docs/progress.md → 看上次做到哪
    ↓
AI: 读取 docs/TASK_BOARD.md → 看当前任务
    ↓
AI: 查询 MemPalace (MCP) → 加载项目上下文和业务规则
    ↓
AI: 读取 graphify GRAPH_REPORT.md → 了解当前代码结构
    ↓
开始工作...
    ↓
工作完成:
  - 更新 docs/progress.md
  - 更新 docs/TASK_BOARD.md
  - 写入 MemPalace 关键变更
  - (可选) 在 Obsidian 创建笔记
  - Git commit → graphify 自动重建
```

## 三、MemPalace 使用规范

### 3.1 配置

项目根目录需创建 `mempalace.yaml`：

```yaml
wing: clm_review_tool
rooms:
  - name: ai_recipe_roadmap
    description: AI菜谱路线图规划与菜谱结构化数据
  - name: frontend
    description: 前端开发记忆：组件、UI规范、用户反馈
  - name: backend
    description: 后端开发记忆：API契约、数据模型、业务规则
  - name: documentation
    description: 文档与交接记录：会议、决策、问题排查
  - name: deployment
    description: 部署与运维记忆：云环境、配置、监控
  - name: general
    description: 通用项目记忆：团队、流程、约定
```

### 3.2 写入时机

| 场景 | 写入内容 | Room |
|------|---------|------|
| 新业务规则确认 | 规则描述 + 触发条件 | backend |
| API 接口变更 | 接口路径 + 请求/响应格式 | backend |
| 数据模型变更 | 表结构变更说明 | backend |
| 部署配置变更 | 环境变量/端口变更 | deployment |
| 问题解决记录 | 问题描述 + 解决方案 | documentation |
| 架构决策 | 决策背景 + 结论 | documentation |

### 3.3 查询时机

| 场景 | 查询内容 |
|------|---------|
| 开始写 API | 查询现有接口契约，避免重复/冲突 |
| 修改数据模型 | 查询现有表结构和关联关系 |
| 处理业务逻辑 | 查询业务规则和边界条件 |
| 部署相关操作 | 查询云环境和配置信息 |


### 4.4 Obsidian 与 docs/ 的区别

| 特征 | Obsidian | docs/ |
|------|----------|-------|
| 归属 | 用户的个人知识库 | 项目的工程文档 |
| 版本控制 | Git 可选 | Git 必须 |
| 双向链接 | 支持 [[wikilink]] | 不支持（标准 Markdown） |
| 图谱视图 | 支持 | 不支持 |
| 主要读者 | 用户自己 | 用户 + AI + 团队成员 |
| 内容性质 | 思考、想法、长期沉淀 | 事实、进度、规范、决策 |

## 五、docs/ 工程文档规范

### 5.1 目录结构

```
docs/
├── 00-PROJECT-INDEX.md      # 项目索引（快速导航）
├── GOVERNANCE.md            # 治理制度（本文档）
├── progress.md              # 进度日志（每日更新）
├── TASK_BOARD.md            # 任务看板（Backlog/Doing/Done）

│   └── ...

```

### 5.2 更新规则

| 文档 | 更新频率 | 触发条件 |
|------|---------|---------|
| progress.md | 每次会话结束 | 工作有进展 |
| TASK_BOARD.md | 任务状态变化时 | 开始/完成任务 |
| RESOURCE.md | 资源变更时 | 新增/修改资源 |
| decisions/ | 做决策时 | 重要技术/产品决策 |

## 六、graphify 使用规范

### 6.1 配置

AGENTS.md 中必须包含：
```markdown
## graphify
- commit/checkout 后自动重建图谱
- 回答架构问题前，先读 graphify-out/GRAPH_REPORT.md
```

### 6.2 使用场景

| 场景 | 用法 |
|------|------|
| 新会话醒来 | 读取 GRAPH_REPORT.md 了解代码结构变化 |
| 改代码前 | 查依赖关系，评估影响范围 |
| 架构问题 | 用图谱代替逐文件读取 |
| 代码审查 | 检查变更是否破坏依赖结构 |

### 6.3 注意事项

- `graphify-out/` 必须被 `.gitignore` 排除
- 每次 commit 后自动重建，保证图谱始终最新
- 图谱大小与代码量成正比，大型项目可能需要几秒

## 七、每次会话的标准流程

### 7.1 会话开始（恢复工作）

```
1. [自动] Memory 注入 → 确认项目身份
2. [自动] 读取 docs/progress.md → 看上次做到哪
3. [自动] 读取 docs/TASK_BOARD.md → 看当前任务
4. [按需] 查询 MemPalace → 加载项目上下文
5. [按需] 读取 graphify GRAPH_REPORT.md → 代码结构
6. [按需] 读取 Obsidian 笔记 → 用户的最新想法/决策
7. 开始工作
```

### 7.2 会话结束（收尾工作）

```
1. 更新 docs/progress.md → 记录做了什么、下一步
2. 更新 docs/TASK_BOARD.md → 移动任务状态
3. [有重要变更时] 写入 MemPalace → 持久化关键信息
4. [有用户决策/思考时] 创建/更新 Obsidian 笔记
5. Git commit → 触发 graphify 自动重建
6. [有重大变化时] 更新 Memory → 阶段/下一步
```

## 八、防遗忘机制（完整）

### 8.1 四层防护

| 防护层 | 防什么 | 怎么防 |
|--------|--------|--------|
| Memory | 忘记项目身份 | 每次会话自动注入 |
| MemPalace | 忘记业务规则和决策 | 语义搜索，跨会话持久 |
| docs/ | 忘记进度和任务 | 版本控制，团队共享 |
| Obsidian | 忘记用户的想法和思考 | 双向链接，长期沉淀 |

### 8.2 自动恢复保证

即使全新会话、全新环境，只要：
1. Memory 中有项目索引 → 知道是哪个项目
2. docs/ 在 Git 中 → clone 后完整恢复
3. MemPalace 配置了 → 查询即可恢复上下文
4. graphify 安装后 → commit 触发重建
