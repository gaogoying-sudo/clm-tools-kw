# ADR 003: 云环境部署策略

**日期:** 2026-04-13  
**状态:** 已批准  
**作者:** 小强  
**优先级:** P1

## 背景

CLM-REVIEW-TOOL 完成本地开发验证后，需要部署到云服务器以支持：
- 一线口味工程师实时访问
- 与公司后台数据库对接
- 飞书消息推送服务
- 7x24 小时定时同步任务

## 决策

### 部署架构

采用 **Docker Compose 单体部署** 方案：

```
┌─────────────────────────────────────┐
│     云服务器 (82.156.187.35)        │
├─────────────────────────────────────┤
│  ┌─────────────┐  ┌──────────────┐  │
│  │   Nginx     │  │ Docker Engine│  │
│  │  (可选)     │  │              │  │
│  └─────────────┘  │ ┌──────────┐ │  │
│                   │ │ Frontend │ │  │
│                   │ │  :8081   │ │  │
│                   │ └──────────┘ │  │
│                   │ ┌──────────┐ │  │
│                   │ │ Backend  │ │  │
│                   │ │  :8001   │ │  │
│                   │ └──────────┘ │  │
│                   │ ┌──────────┐ │  │
│                   │ │  MySQL   │ │  │

│                   │ └──────────┘ │  │
```

### 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 容器化 | Docker + Docker Compose | 本地已验证，部署一致性好 |
| Web 服务器 | Nginx (可选) | 反向代理、HTTPS、静态资源缓存 |
| 数据库 | MySQL 8.0 | 与公司后台一致，便于数据同步 |
| 进程管理 | Docker restart policy | 自动重启，无需 systemd |
| 日志管理 | Docker logs + 文件轮转 | 简单够用，后期可接 ELK |

### 部署策略

**阶段 1: 基础部署（当前）**
- Docker Compose 直接部署
- 端口直连（8081/8001）
- 基础监控（健康检查 + 日志）

**阶段 2: 生产加固（后续）**
- Nginx 反向代理 + HTTPS
- 数据库主从备份
- 监控系统（Prometheus + Grafana）
- CI/CD 自动化

### 端口规划

| 服务 | 容器端口 | 主机端口 | 对外 |
|------|---------|---------|------|
| Frontend | 80 | 8081 | 是 |
| Backend | 8001 | 8001 | 是（API） |
| MySQL | 3306 | 3307 | 否（仅内网） |

### 安全策略

1. **数据库隔离** - MySQL 不暴露公网端口
2. **API 鉴权** - ADMIN_TOKEN 保护管理后台
3. **飞书安全** - DRY_RUN 默认开启，白名单机制
4. **SSH 密钥** - 仅允许密钥登录，禁用密码
5. **防火墙** - 仅开放 80/443/22 端口

### 数据持久化

```yaml
volumes:
  - mysql-data:/var/lib/mysql
  - ./backend/app:/app/app  # 开发模式
  - ./docs:/opt/docs       # 文档挂载
```

### 备份策略

- **数据库备份:** 每日凌晨 2 点 mysqldump
- **备份保留:** 7 天本地 + 30 天远程
- **恢复演练:** 每月一次

## 替代方案

### 方案 A: Kubernetes 部署

**优点:**
- 高可用、自动扩缩容
- 服务网格、细粒度流量控制

**缺点:**
- 运维复杂度高
- 资源开销大
- 当前规模不必要

**结论:** 过度设计，暂不采用

### 方案 B: 云原生 PaaS（如 Railway/Render）

**优点:**
- 零运维
- 自动 HTTPS
- 内置 CI/CD

**缺点:**
- 成本较高
- 国内访问速度慢
- 数据合规风险

**结论:** 不适合国内企业环境

### 方案 C: 传统虚拟机部署

**优点:**
- 直接控制
- 无容器开销

**缺点:**
- 环境一致性差
- 回滚困难
- 资源利用率低

**结论:** 已被 Docker 方案替代

## 用户需手动配置项

以下信息需要用户提供并填写到 `.env.prod`：

### 必须配置
```bash
MYSQL_ROOT_PASSWORD=<强密码>
MYSQL_PASSWORD=<强密码>
ADMIN_TOKEN=<管理后台访问令牌>
SOURCE_DB_HOST=<公司后台数据库主机>
SOURCE_DB_USER=<公司后台数据库用户名>
SOURCE_DB_PASSWORD=<公司后台数据库密码>
```

### 可选配置
```bash
FEISHU_DRY_RUN=false        # 关闭演练模式，启用真实推送
FEISHU_TEST_MODE=false      # 关闭测试群模式，启用生产群
```

## 验证标准

- [ ] 所有 Docker 容器健康检查通过
- [ ] 前端页面可访问（HTTP 200）
- [ ] 后端 API 健康检查通过
- [ ] 数据同步 API 正常工作
- [ ] 飞书消息推送测试成功（先 DRY_RUN）
- [ ] 定时任务正常执行
- [ ] 日志正常无错误

## 回滚方案

1. **代码回滚:** `git checkout <PREVIOUS_COMMIT> && docker compose up -d`
2. **数据回滚:** 从备份恢复 MySQL 数据库
3. **配置回滚:** 恢复 `.env.prod` 旧版本
