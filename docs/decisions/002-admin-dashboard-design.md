# 002. 管理后台看板重构

**日期：** 2026-04-12  
**状态：** 已采纳

## 背景
旧管理后台是手机优先的单页应用，所有功能挤在一个页面内。管理端需要桌面级看板，支持多页面、多维度查询、原始数据保留。

## 决策

### 5 页 + 登录架构

| 页面 | 路由 | 数据粒度 | 核心功能 |
|------|------|---------|---------|
| 登录 | /login | - | 用户名密码 + 角色路由 |
| 数据看板 | /dashboard | 聚合指标 | 全局概览、趋势图、热力图 |
| 数据检索 | /search | 工程师/会话级 | 多条件搜索、缓存命中、时间线 |
| 原始问答 | /qa-records | 问题/回答级 | 三层数据保留、全文检索、导出 |
| 经验沉淀 | /insights | 经验条目级 | 审核流程、分类筛选 |
| 系统管理 | /settings | 配置/权限级 | 用户管理、模板管理、花名册 |

### 技术选型
- 前端：React + TailwindCSS 桌面端布局（侧边栏 + 内容区）
- 组件化：每个页面拆为独立组件 + 共享 UI 组件
- 后端：JWT token 鉴权 + 角色中间件
- 缓存：本地 SQLite 缓存已检索结果（避免重复查询）

### 权限模型
| 角色 | 可访问页面 |
|------|-----------|
| 超级管理员 | 全部 5 页 |
| 管理员 | dashboard + search + qa-records + insights |
| 数据分析师 | dashboard + search（只读）+ qa-records（只读） |

### 检索缓存逻辑
1. 检查是否当天数据 → 是则标记"进行中"，不直接用
2. 检查本地缓存是否有该检索条件组合
3. 有则直接返回，无则后端查询 + 写入缓存

### 原始数据保留策略
- 不可删除，只可归档
- 三层保留：raw_input / transcribed_text / structured_result
- 支持按日期范围导出 JSON/CSV

## 理由
- 手机端排版无法承载多维度管理需求
- 原始问答数据需要独立入口，不与其他功能混淆
- 缓存层减少重复查询，提升响应速度
- 角色权限是管理系统的刚需

## 影响
- 前端需要完全重构（从内联样式到 TailwindCSS + 组件化）
- 后端需要新增 JWT 鉴权中间件
- 需要新增缓存层 API
- 旧的 admin 页面路由废弃
