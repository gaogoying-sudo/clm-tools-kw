## 一、项目全貌理解

### 1.1 一句话概括
让一线口味工程师每天录菜结束后，通过飞书收到当日任务复盘问卷，填写后自动沉淀为可检索的经验/失败/调优知识库。

### 1.2 核心业务链路
```
公司后台DB (录菜记录) 
    → DateUse数据同步 (清洗/映射) 
    → CLM-Tools MySQL (结构化)
    → 生成当日会话 + 动态问题流
    → 飞书推送工程师
    → 工程师填写提交
    → 经验候选入库 + 管理端审核
```

### 1.3 三团队现状（已合并为小强一人）

| 团队 | 原职责 | 代码位置 | 状态 |
|------|--------|---------|------|
| CLM-Tools (Cursor) | review-app + worker 业务逻辑 | ~/software project/cursor/CLM-Tools/dailyReport/clm-review-tool/ | 有完整前后端+DB模型 |
| DateUse (Codex) | data-sync 数据同步 | ~/software project/DateUse/app/data-sync/ | 有工程师映射+DB连接逻辑 |
| 小厨 (Hermes) | 协调/部署/文档 | 云服务器 82.156.187.35 | 环境已初始化，MySQL运行中 |

### 1.4 核心约束（来自交接文档）
- review-app **不直连公司源数据库**，只读 DateUse 写入的结构化结果
- 回传数据必须在 MySQL 可追溯（不能只留在飞书/日志）
- 飞书推送必须安全封闭（dry-run + 测试群 + 白名单）
- 数据库分层：raw_* / core_* (DateUse) + biz_* / asset_* (CLM-Tools)

## 二、现有代码资产分析

### 2.1 CLM-Tools (后端) —— FastAPI + SQLAlchemy + MySQL

**数据库模型（7张表）：**
1. `engineers` — 工程师信息（含飞书user_id映射）
2. `sync_tasks` — 当日录菜任务（含功率轨迹、投料时序、调整记录等过程数据）
3. `daily_sessions` — 每日回传会话（含summary_snapshot、root_event_type、问题流版本）
4. `session_tasks` — 会话-任务关联（多对多）
5. `questions` — 问题模板（支持触发规则、分支逻辑、条件可见）
6. `answers` — 回答（三层存储：raw_input → transcribed_text → structured_result）
7. `experience_candidates` — 经验候选（8种分类，4种审核状态）

**API路由：**
- `sessions.py` — 同步今日数据、创建会话、飞书推送
- `answers.py` — 提交回答、自动分类入池
- `admin.py` — 管理端（查看/审核/模板管理）

**核心引擎：**
- `sync_service.py` — 从数据源拉取当日任务（当前mock模式）
- `question_engine.py` — 信号检测 + 动态问题选择 + 触发逻辑
- `feishu.py` — 飞书消息发送封装

**技术栈：** FastAPI + SQLAlchemy + APScheduler + Pydantic + MySQL 8.0

### 2.2 CLM-Tools (前端) —— React + Vite

- 单页应用，基于 hash 路由 (#/ / #/fill/:id / #/admin)
- 三个视图：工程师首页 → 填写流程 → 管理后台
- 管理后台含4个Tab：今日总览、历史记录、经验候选、问题管理
- 内联样式，无CSS框架，移动端优先设计
- 技术栈：React (JSX) + Vite

### 2.3 DateUse (data-sync)

- `engineers.py` — 36位工程师花名册（含别名）
- `db.py` — 公司源DB连接（pymysql，只读事务）
- `mapper.py` — 工程师身份映射逻辑（多源匹配、admin_id桥接）
- `config.py` — 配置管理
- `api_server.py` — API服务
- `main.py` — 主入口

**核心能力：** 从公司后台 btyc 数据库的 ums_admin / auth_user 表中提取账户池，与工程师花名册做模糊匹配，桥接 admin_id。
